# Release Process

Self-Harness releases must preserve the paper-fidelity and auditability
contracts before publishing artifacts.

## Versioning

- Patch version: documentation, tests, or implementation changes that do not
  change public APIs or audit schema.
- Minor version: additive public API changes or additive audit schema fields.
- Major version: breaking public API changes or breaking audit schema changes.

Audit schema changes must follow `docs/architecture/audit_schema_policy.md`.
Breaking schema changes require a migration shim before the new major version
becomes the default writer version.

Use `self-harness audit-migrate SOURCE --out DEST` to create an upgraded copy of
an older audit tree. Migration is upgrade-only release/operator tooling: it
does not mutate the source audit, rotate canonical readiness hashes, or claim
benchmark reproduction. Use `--target-major` for explicit breaking-schema
migration checks and `--allow-lossy` only with reviewed drop-only transforms.
See `docs/operations/audit_migration.md`.

## Release Gate

Before tagging:

```bash
make check
make readiness
make audit-verify
make audit-verify-live
make operator-check
make readiness-matrix
make container-preflight
make readiness-drift-check
make reproduction-readiness-artifact-shape-lint ARTIFACT_DIR=dist/reproduction-artifacts
make reproduction-readiness-check
make operator-promotion-check
make operator-policy-binding-check
make attestation-check
make migration-check
make release-candidate-evidence
make build
make release-smoke
```

The readiness gate runs the paper-fidelity invariant suite and verifies the
canonical deterministic audit hash in `tests/fixtures/canonical_audit_hash.txt`.
CI must pass on Python 3.11, 3.12, and 3.13 before a release tag is considered
ready.
The canonical audit hash includes derived `trajectory.jsonl` bytes generated by
`self-harness audit-trajectory`.

`make audit-verify` is an offline audit-integrity gate. It creates a
deterministic local audit under `dist/`, writes a structured verification report,
and checks manifest, lineage, harness hashes, proposal/evaluation rows, split
totals, and paper-fidelity leakage boundaries. It does not mutate audit
artifacts, execute tasks outside the deterministic local gate, contact external
systems, or claim benchmark reproduction. See `docs/source/audit_integrity.md`.

`make audit-verify-live` exercises the signed live-audit provenance seam with
an offline fixture. It generates a synthetic Harbor artifact tree, ingests it
into an audit directory, signs provenance for a live-shaped Harbor audit
artifact, and writes `dist/self-harness-audit-verify-live.json`. Real
production use supplies operator-owned live artifacts to
`self-harness audit-verify-live`; the command emits `mode:"live"` only when the
replay audit verifier, provenance signature, live artifact shape, and task-id
binding checks all pass. See `docs/operations/audit_verify_live.md`.

## Vulnerability Policy Gate

`make release-smoke` includes `make vuln-check`. This gate inspects the built
wheel's runtime `Requires-Dist` dependencies, runs `pip-audit` when runtime
dependencies are present, and writes `dist/self-harness-vuln-report.json`.
Findings fail closed unless they are explicitly allowed by an operator-owned
policy file.

Use `VULN_POLICY=path/to/vulnerability-policy.json make vuln-check` to apply a
policy. Policy entries must name the package, vulnerability id, status,
justification, and expiry date. Expired, retired, rejected, malformed, or
missing entries do not satisfy the gate. The policy file is release/operator
material, not an audit artifact, and does not affect paper-fidelity readiness
hashes or reproduction-claim semantics.

Operators can evaluate container image scanner reports with the same policy
format. Generate a Trivy JSON report outside Self-Harness, then evaluate it
offline:

```bash
trivy image --format json --output image-vulns.json registry.example/verifier@sha256:<digest>
python scripts/vuln_check.py \
  --format trivy \
  --audit-json image-vulns.json \
  --image-policy path/to/image-policy.json \
  --freshness-policy path/to/freshness-policy.json \
  --policy path/to/vulnerability-policy.json
```

This checks scanner findings against policy; it does not pull images, run
Docker, contact registries, or claim benchmark reproduction. When
`--image-policy` is supplied, the Trivy report must include
`Metadata.RepoDigests` and at least one `image@sha256:<digest>` entry must match
an active image-policy entry. When `--freshness-policy` is supplied, the Trivy
report must include top-level `CreatedAt` or `Metadata.CreatedAt` and satisfy
the operator-owned `max_age_days` and/or `not_before` rule. Missing, malformed,
future-dated, or stale scanner timestamps fail closed with exit code 2.

`make scanner-check` exercises the scanner execution orchestrator in dry-run
mode. It verifies deterministic Trivy command construction without requiring
Trivy, Docker, registries, or a scanner database. It also dry-runs the Trivy DB
update command and replays a checked-in Trivy report fixture against checked-in
scanner DB metadata and DB freshness policy, proving the DB lifecycle path
offline. Operators with Trivy installed can use `scripts/scanner_run.py
--replay` for deterministic fixture evaluation or omit `--dry-run`/`--replay`
to run a live scanner preflight and scan. Live scanner outputs remain
release/operator material; they do not affect audit schemas, readiness hashes,
or benchmark reproduction claims.

Scanner and scanner-DB update commands accept `--db-registry-config` for
operator-owned Trivy registry auth files. The path is included in dry-run
command traces; file contents are never read or written by dry-run checks and
must not be included in release artifacts or logs.

`make harbor-discovery-check` exercises pre-run Harbor image discovery in
dry-run and replay modes. It validates Harbor API request construction and
strict fixture parsing of `sha256:<digest>` artifact responses without
contacting a live Harbor host. Discovery output is release/operator material
used to feed image-policy decisions; it is not benchmark reproduction evidence.

`make operator-check` validates a versioned operator policy bundle and runs a
consolidated offline preflight over policy parsing, trusted public keys,
scanner command construction, scanner DB update command construction, registry
config path handling, and Harbor discovery replay. It does not run Trivy,
download scanner databases, contact Harbor, contact registries, publish to
PyPI, invoke Sigstore, or claim benchmark reproduction. See
`docs/operations/operator_bundle.md`.

`make readiness-matrix` validates the checked-in live-dependency catalog and
writes `dist/self-harness-readiness-matrix.json`. The report records whether
Harbor, Docker, Trivy, Sigstore, PyPI, model, scanner mirror, and signer
dependencies are blocked, optional, or provisioned. It does not probe the
environment, run live tools, mutate audit artifacts, rotate readiness hashes,
or claim benchmark reproduction. `make release-candidate-evidence` requires
this report and fails closed when it is missing, malformed, failed, or claims
reproduction. A valid report may still set `live_execution_blocked` to
`true` until operators provision the live dependencies. See
`docs/operations/readiness_matrix.md`.

`make readiness-drift-check` cross-checks the readiness catalog against the
existing offline preflight artifacts and writes
`dist/self-harness-readiness-drift.json`. Blocked and optional entries are
advisory. A provisioned reproduction-relevant dependency fails closed unless
its declared preflight surface is present, well-formed, passing, and has no
failed required checks. This gate does not run live tools, mutate audits,
rotate canonical readiness hashes, or claim benchmark reproduction. `make
release-candidate-evidence` requires its JSON report. The PyPI entry is bound
to the installed-wheel `release_smoke` surface, but remains `blocked`; the
smoke status proves offline installability only, not trusted publishing. See
`docs/operations/readiness_matrix.md`.

`make readiness-promotion-check` compares `READINESS_BASELINE_CATALOG` to
`READINESS_CANDIDATE_CATALOG` and writes
`dist/self-harness-readiness-promotion.json`. It is read-only and the default
release path includes it as an advisory release-candidate evidence gate;
provisioned reproduction-relevant transitions must be backed by the same
surface evidence that readiness drift requires. It does not run live tools,
mutate catalogs, rotate readiness hashes, or claim benchmark reproduction.

`make container-preflight` writes
`dist/self-harness-container-preflight.json` in offline mode. It checks Docker
CLI discovery and records Docker daemon and image checks as skipped, so it is
advisory while the Docker row remains blocked. A future provisioned Docker row
requires operator-owned live container-preflight evidence. The Sigstore row is
similar: structural attestation evidence is advisory while blocked; provisioned
Sigstore readiness requires `verify-attestation --backend sigstore` with
`cryptographic_valid: true`.

`make model-backend-preflight` writes
`dist/self-harness-model-backend-preflight.json` for the three paper model
backends. Dry-run mode is the default and does not contact providers; live mode
requires explicit operator endpoint and credential environment. Readiness drift
ingests the report when present, but the command is not part of the default
release path and does not claim benchmark reproduction. See
`docs/operations/model_backend_preflight.md`.

`make reproduction-readiness-check` maps the paper's live benchmark-readiness
requirements to the readiness matrix and artifact evidence, then writes
`dist/self-harness-reproduction-readiness.json`. A valid current report may set
`reproduction_ready` to `false`; that is expected while live Harbor, Docker,
paper model backend, network-control, artifact-ingest, PyPI, and Sigstore
material is absent.
`make reproduction-readiness-artifact-shape-lint` validates an operator-supplied
artifact directory against the required live evidence shapes before the hard
reproduction gate is attempted.
`make reproduction-bundle-check` builds a deterministic reproduction bundle
manifest from operator-supplied artifact files, signs the exact manifest bytes,
and verifies the signed bundle. It requires explicit bundle id, created-at,
operator-label, source metadata, and signing material; it is standalone and is
not part of the default release path.
`make reproduction-readiness-bundle-verify` validates the signed operator
bundle manifest that binds the same artifact set by path, byte size, and
SHA-256 digest. It is required only by the hard reproduction path.

`make capture-manifest-build`, `make capture-rehearsal`,
`make capture-manifest-check`, and `make capture-manifest-diff-check` exercise
the operator live-evidence capture manifest authoring contract, signed plan
verification, synthetic bundle rehearsal, and plan-vs-realized bundle diff.
Capture manifests and rehearsal bundles are signed pre-capture operator
material, not live evidence; the rehearsal and diff cannot satisfy reproduction
requirements by themselves. See `docs/operations/capture_manifest.md`.
`make capture-extract-check` exercises offline post-capture extractors that
turn operator raw outputs into strict live artifact-class JSON shapes before
bundle packaging, including the fixed 64-case split manifest and the fixed
paper model/evaluator/tool-budget/round-count/proposal-width protocol
declaration plus proposer request-log/context manifests. Live audit and
two-repeat evaluation artifacts carry the fixed protocol artifact hash so
bundle verification can reject protocol drift. It is
standalone and does not contact live services. See
`docs/operations/capture_extract.md`.
`make capture-admit-check` exercises the offline admission orchestrator that
binds raw inputs, extracted artifacts, bundle verification, and optional
readiness evaluation into one hash-stable report. It is standalone and does not
contact live services. See `docs/operations/capture_admit.md`.
`make audit-verify-live` is the matching post-capture audit verification seam:
it can produce the live-shaped `audit_verify_report` artifact class for
reproduction readiness, but only from signed existing evidence.

The standalone script exits non-zero for not-ready reports, while the Make
target preserves the JSON for inspection. See
`docs/operations/benchmark_reproduction_readiness.md`.

`make operator-promotion-check` verifies an operator policy promotion manifest
and a detached Ed25519 signature sidecar generated with temporary local test
key material. The private key is kept in a `mktemp` directory; only the public
key, signature sidecar, and preflight report are written under `dist/`.
Promotion manifests bind release-policy files to SHA-256 digests and monotonic
lifecycle states. They do not run Trivy, download scanner databases, contact
Harbor, contact registries, publish to PyPI, invoke Sigstore, mutate audit
artifacts, rotate readiness hashes, or claim benchmark reproduction. See
`docs/operations/operator_promotion.md`.

`make operator-policy-binding-check` cross-checks the operator bundle paths
against active promotion manifest entries and their SHA-256 digests. It catches
release-policy drift where the bundle and promotion manifest are individually
valid but point at different operator material. This gate is offline and does
not mutate audit artifacts, rotate readiness hashes, contact live services, or
claim benchmark reproduction. `make release-candidate-evidence` requires its
JSON report. See
`docs/operations/operator_policy_binding.md`.

`make release-candidate-evidence` aggregates the offline gate outputs into
`dist/self-harness-release-candidate-evidence.json`. The decision is `blocked`
when any required gate artifact is missing, malformed, failed, or claims
benchmark reproduction. This evidence document is release/operator material,
not benchmark reproduction evidence. The Makefile production path includes the
generated readiness matrix report as required release evidence and the
generated readiness drift report as required release evidence. The generated
readiness promotion report is advisory release evidence. The generated
reproducible build report is required release evidence. The generated
structural attestation report remains optional release evidence. See
`docs/operations/release_candidate_evidence.md`.

`make release-candidate-evidence-reproduction` opts into the hard benchmark
reproduction gate and writes
`dist/self-harness-release-candidate-evidence-reproduction.json`. It is
expected to remain blocked until the reproduction readiness report is true and
a signed reproduction bundle report is present.

## Local Release Smoke Gate

`make reproducible-build-check` is required before publishing. It runs
`make build` with the project `SOURCE_DATE_EPOCH`, rebuilds the wheel from the
source distribution without build isolation, and blocks if the rebuilt wheel's
filename or SHA-256 differs from the wheel in `dist/`. The report is written to
`dist/self-harness-reproducible-build.json` and consumed by release-candidate
evidence. This is package supply-chain evidence; it does not publish to PyPI,
validate trusted publishing, or claim benchmark reproduction.

`make release-smoke` is required before tagging. It builds the wheel, installs
that wheel into a fresh isolated virtual environment, imports the stable public
API, runs the installed `self-harness` CLI, writes `trajectory.jsonl`, and
verifies that the installed-wheel audit hash matches
`tests/fixtures/canonical_audit_hash.txt`. This catches packaging and console
script failures that source-tree tests or editable installs can miss.
The installed-wheel smoke step writes
`dist/self-harness-release-smoke.json`, a deterministic
`release_smoke_status/1.0` report consumed by readiness drift. That report is
offline installability evidence only; it does not publish to PyPI or validate
trusted-publishing configuration.

`make sbom` can generate a local CycloneDX SBOM when the release dependencies
are installed. Generate it before `make provenance` when the SBOM should be
included in the release provenance manifest. Missing SBOM tooling does not
affect the offline smoke gate.

## Release Provenance Manifest

`make provenance` writes a deterministic release provenance manifest under
`dist/` after the wheel and source distribution are built. The manifest records
package metadata, build backend metadata, best-effort git source information,
and SHA-256 hashes plus byte sizes for the wheel, source distribution, and
optional SBOM if present.

`make release-smoke` requires this manifest and verifies every referenced
artifact before installing the wheel into an isolated environment. A missing,
malformed, or hash-mismatched provenance manifest is a release-blocking error.
The provenance manifest is content-addressed release material. Sigstore and
PyPI attestation pre-validation is a separate release/operator trust boundary.

## Release Attestation Pre-Validation

`make attestation-check` builds a synthetic local PyPI attestation envelope for
the built wheel and verifies it with the structural backend. The report is
written to `dist/self-harness-attestation.json` and is included by the Makefile
release-candidate evidence target.

The structural backend checks local envelope shape, material SHA-256 binding,
signature field presence, certificate chain presence, transparency-log field
presence, and certificate issuer/SAN allowlists from an operator-owned trust
root. It does not contact Fulcio, Rekor, PyPI, Sigstore, Harbor, Docker,
registries, scanners, models, or cloud providers. It never claims
cryptographic validity; `cryptographic_valid` is `null`.

For real release signing or publishing, use an operator-supplied Sigstore/PyPI
workflow and treat the structural report as pre-validation only. The optional
`sigstore` extra enables offline cryptographic verification of canonical
Sigstore bundle material when the attestation trust root points to a full
Sigstore client trust config or trusted-root JSON file. Cryptographic
verification remains release/operator material and still does not claim
benchmark reproduction. See `docs/operations/attestations.md`.

## Release Provenance Signature

Release maintainers can add a detached Ed25519 signature sidecar over the exact
manifest file bytes:

```bash
make provenance
python scripts/sign_provenance.py \
  --manifest "$(ls dist/*-provenance.json)" \
  --private-key keys/release-provenance.ed25519 \
  --public-key keys/release-provenance.ed25519.pub
python scripts/verify_provenance_signature.py \
  --manifest "$(ls dist/*-provenance.json)" \
  --signature "$(ls dist/*-provenance.json.sig)" \
  --public-key keys/release-provenance.ed25519.pub
```

`make provenance-sign` wraps the same flow for CI or operator scripts. Set
`RELEASE_PROVENANCE_KEY` and optionally `RELEASE_PROVENANCE_PUBLIC_KEY`,
`RELEASE_PROVENANCE_KEY_ID`, `RELEASE_PROVENANCE_PROVIDER`,
`RELEASE_PROVENANCE_FINGERPRINT`, and `RELEASE_PROVENANCE_PASSPHRASE_ENV`.
Alternatively set `RELEASE_PROVENANCE_EXTERNAL_SIGNER` to use the external
signer protocol without exporting private key material.

`make release-smoke` does not require a signature by default, but it verifies a
sibling `*.sig` sidecar when present. Passing `--provenance-signature` to
`scripts/release_smoke.py` makes the sidecar explicit and fail-closed. Passing
`--provenance-public-key` verifies the sidecar against trusted release key
material; otherwise the smoke gate only checks sidecar self-consistency.

Provenance signatures are release material, not audit artifacts. They do not
change the audit schema, task corpus schema, benchmark provenance, or
reproduction-claim rules. Private keys, passphrases, external-signer stderr,
and secret-manager material must never be written to sidecars, artifacts, or
release notes.

Signed production corpora should be validated with:

```bash
self-harness validate-tasks path/to/corpus.json --require-corpus-signature path/to/corpus.ed25519.pub
```

Generate offline signing keys and sign production corpora with:

```bash
self-harness corpus-keygen --out keys/corpus.ed25519
self-harness corpus-fingerprint --public-key keys/corpus.ed25519.pub
self-harness corpus-sign --corpus path/to/corpus.json --private-key keys/corpus.ed25519 --out path/to/corpus.signed.json
```

Production private keys can be encrypted with an explicit noninteractive
passphrase source:

```bash
self-harness corpus-keygen --out keys/corpus.ed25519 --passphrase-env CORPUS_KEY_PASSPHRASE
self-harness corpus-sign \
  --corpus path/to/corpus.json \
  --private-key keys/corpus.ed25519 \
  --passphrase-env CORPUS_KEY_PASSPHRASE \
  --out path/to/corpus.signed.json
```

Trusted public keys can be maintained in a portable keyring:

```bash
self-harness corpus-keyring add \
  --keyring keys/corpus.keyring.json \
  --corpus-id local-smoke \
  --public-key keys/corpus.ed25519.pub
self-harness validate-tasks path/to/corpus.signed.json --require-corpus-keyring keys/corpus.keyring.json
```

Corpus signing keys are operator-held release material. Private keys must never
be committed, embedded in task corpora, written to audit artifacts, or pasted
into release notes. Do not rotate corpus signing keys silently. Key rotations
must be recorded in release notes with the affected corpus IDs and the new
public-key fingerprint.
Encrypted private keys use cryptography's PKCS8 best-available PEM envelope.
Passphrases must come from a secret manager, environment variable, or protected
file and must never be written to release notes, corpus JSON, keyring JSON, or
audit artifacts.

When signing material is held by KMS, HSM, YubiKey, or platform-keychain
tooling, prefer the external signer protocol instead of exporting key material
to this project:

```bash
self-harness corpus-sign \
  --corpus path/to/corpus.json \
  --external-signer "python ops/kms_corpus_signer.py" \
  --signer-provider aws-kms \
  --key-id alias/self-harness-corpus \
  --fingerprint <64-hex-public-key-fingerprint> \
  --out path/to/corpus.signed.json
```

External signer wrappers must sign the canonical payload received on stdin and
must not print key material. Record the provider, key id, public-key
fingerprint, and affected corpus IDs in release notes or an internal trust
registry before promoting the signed corpus.

`scripts/example_external_signer.py` is a local-only protocol reference for
testing the external signer boundary with an Ed25519 PEM key path supplied via
environment variable. It is not a production KMS, HSM, YubiKey, or
secret-manager wrapper.

Provider packages that supply KMS, HSM, OAuth/OIDC, registry, or secret-manager
integrations should implement the contracts in `self_harness.providers`. The
in-repository static providers are for tests and local dry-run demos only; do
not load production secret material into them. See
`docs/operations/provider_seams.md`.

Corpus keyrings are also operator-held release material. They may embed public
keys and labels, but must never embed private keys. During rotations, keep the
new key `active` and move superseded keys to `retired` or `revoked`; only
`active` keyring entries satisfy validation gates.

## Canonical Audit Hash Rotation

Rotate the canonical hash only when the audit byte layout intentionally changes.
Rotation requires:

- a schema-minor bump for additive layout changes, or a schema-major bump plus a
  migration shim for breaking layout changes;
- an entry in `docs/architecture/schema_changelog.md` explaining the rotation;
- a regenerated `tests/fixtures/canonical_audit_hash.txt` from the deterministic
  demo run.

Any maintainer may rotate the hash when those conditions are met.

## Tagging

- Release tag format: `vX.Y.Z`.
- Release-candidate dry-run tag format: `vX.Y.Z-rc.N`.
- `-rc.` tags run the full release workflow but skip TestPyPI and PyPI
  publication.

## Publishing

The GitHub release workflow uses PyPI trusted publishing through OIDC. Repository
owners must configure trusted publishers in TestPyPI and PyPI before publishing
non-RC tags.

Release notes should summarize the relevant README, schema changelog, and
architecture changelog entries.

Optional provider adapters must remain optional. Before release, CI's core
import job must prove `import self_harness` works without installing provider
extras, and the Anthropic plus paper-backend adapter contract tests must pass
without real API keys or live provider endpoints.

Benchmark reports must keep `reproduction_claimed=false` unless the underlying
audits include complete model, Harbor, dataset, corpus, container, and split
provenance. Placeholder values such as `unknown-live` are acceptable for
readiness reports, not reproduction claims.

Harbor artifact ingests created by `self-harness harbor-ingest` start with
`harbor_artifact_validation_status` set to `candidate` or `partial`. Do not
mark an artifact layout `validated` in release notes until a real preserved
Harbor run has been inspected and the layout is documented.
